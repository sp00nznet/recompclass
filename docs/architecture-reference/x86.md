# x86 (DOS / Xbox / Win32) Quick Reference

## Overview

x86 is a CISC architecture spanning decades of PC and console history. For recompilation purposes, the key variants are:

- **16-bit Real Mode (8086/80286):** DOS games, early PCs
- **32-bit Protected Mode (80386+):** Win32 applications, original Xbox (Pentium III)

Variable-length instruction encoding (1-15 bytes). Little-endian.

## Segment:Offset Addressing (Real Mode)

Physical address = (Segment << 4) + Offset

Example: CS=0x1000, IP=0x0200 --> Physical 0x10200

The 20-bit address space (1 MB) wraps at 0xFFFFF unless A20 is enabled.

## x87 FPU

Stack-based floating point with 8 x 80-bit registers (ST(0)-ST(7)).

| Instruction | Description                    |
|-------------|--------------------------------|
| FLD         | Push value onto FPU stack      |
| FST, FSTP   | Store (and pop) from stack    |
| FADD, FSUB  | Add/subtract                  |
| FMUL, FDIV  | Multiply/divide               |
| FCOM, FCOMP | Compare                       |
| FXCH        | Exchange ST(0) with ST(n)     |
| FILD, FISTP | Integer load/store            |

## Xbox (Original) Notes

The original Xbox uses an Intel Pentium III (733 MHz, 32-bit x86) with:
- 64 MB unified RAM
- Standard Win32 subset (no full Windows kernel)
- NV2A GPU (custom NVIDIA)
- No segment trickery -- flat 32-bit addressing

## Recompiler Pain Points

- **Variable instruction length:** Instructions are 1-15 bytes. Correct disassembly requires sequential decoding; you cannot jump into the middle of an instruction (but self-modifying code can create apparent mid-instruction entry points).
- **Segment addressing (real mode):** Every memory access is segment-relative. Segment registers change frequently. Far calls/jumps change CS. Recompilers must track or flatten segment:offset pairs.
- **EFLAGS complexity:** Nearly every ALU instruction modifies flags, but each instruction modifies a different subset. Lazy flag evaluation is almost mandatory for performance.
- **x87 FPU stack:** The stack-based FPU model does not map cleanly to SSE or host FPU registers. Stack depth tracking and ST(n) renaming add complexity.
- **Self-modifying code:** Common in DOS-era copy protection and some games. x86 has no inherent separation between code and data memory.
- **REP-prefixed string operations:** MOVS/STOS/CMPS with REP can process large blocks. They are interruptible and restartable, which complicates recompilation.
- **Interrupt handling (DOS):** Software interrupts (INT xxh) invoke BIOS/DOS services that must be emulated or stubbed out.
- **Partial register writes:** Writing to AL does not zero AH (unlike x86-64 behavior with 32-bit writes). This creates hidden dependencies.
